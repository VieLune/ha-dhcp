# AC 高可用功能说明

## 1. 主备状态

系统默认部署在 Ubuntu。运行时只看本机是否持有 VIP：

- 本机持有 VIP：主节点，可响应 DHCP，可处理写入类 API。
- 本机不持有 VIP：备节点，不响应 DHCP，只做只读查询、健康检查和本地镜像同步。
- 网络分区期间允许两台 AC 各自持有 VIP 并继续服务。

## 2. DHCP 服务

DHCP 服务只在以下条件同时满足时响应：

- DHCP 已启用。
- AC 配置已加载。
- 本地租约镜像已加载。
- 本机实际持有 VIP。

Hazelcast 成员不足不会直接阻断 DHCP。分区期间两侧可能产生不同租约视图，恢复后以 VIP 持有侧优先合并；被弃用的租约后续请求返回 DHCPNAK。

## 3. AC 配置

配置接口：

```text
GET /internal/ac/config
PUT /internal/ac/config
```

`application.yml` 只作为初始默认值。启动时如果 H2 中存在 `ac_config.effective`，数据库配置覆盖默认值。`PUT` 成功后立即持久化到 H2，并发布到 Hazelcast `ac:config`。

第一版配置范围为 DHCP + HA 相关字段，包括 `ha.ip`、`ha.interface-name`、DHCP 地址池、租期、DNS、排除地址和保留地址等。

## 4. 边缘设备

边缘设备包括遥测仪器、穿戴设备等。第一版只维护 SN 与 IP：

```text
GET /internal/edge-devices
GET /internal/edge-devices/{sn}
PUT /internal/edge-devices/{sn}
```

写入示例：

```json
{ "ip": "192.168.56.210" }
```

`PUT` 仅允许当前 VIP 持有节点执行。SN trim 后作为 key，IP 必须是 IPv4，最终视图中 IP 唯一。写入成功后同步保存到 H2 `edge_device` 和 Hazelcast `edge:devices`。

## 5. 分区恢复表现

| 场景 | 期望行为 |
|---|---|
| 单节点只看到自己 | 尝试获取 VIP，成功后继续 DHCP 和写入 API。 |
| 两节点网络分区 | 两侧可各自持有 VIP，允许双主。 |
| 网络恢复 | 先收敛 VIP，再由 VIP 持有侧生成最终数据视图。 |
| DHCP 租约冲突 | 主侧优先，冲突请求后续 DHCPNAK。 |
| 边缘设备 SN/IP 冲突 | 主侧优先，非冲突记录补并。 |
| AC 配置冲突 | 主侧完整配置覆盖非主侧。 |

## 6. 运维检查

常用检查：

```bash
ip addr show
curl -i http://<VIP>:<HTTP_PORT>/internal/ha/status
curl -i http://<VIP>:<HTTP_PORT>/internal/ac/config
curl -i http://<VIP>:<HTTP_PORT>/internal/edge-devices
```

状态接口需要关注 `hasVip`、`vipAddress`、`leaseMirrorLoaded`、`configLoaded`、`leaseAuthorityMembers` 和 `canServeDhcp`。

## 7. 验收点

- `ha.ip` 配置后作为 VIP；未配置时使用 Ubuntu 第一块物理网卡 IPv4。
- 非 VIP 节点拒绝 AC 配置和边缘设备写入。
- API 修改的配置重启后仍由 H2 加载。
- 分区期间两侧可服务，恢复后只保留一个 VIP 持有者。
- DHCP、边缘设备和 AC 配置按主侧优先策略恢复。
