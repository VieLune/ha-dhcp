# AC 高可用功能说明

## 1. 功能概览

本方案提供 AC 管理端 HTTP/API 高可用、DHCP 单主热备、租约同步、健康检查和业务网络副作用控制。

运行形态：

```text
AC-1：业务节点，正常情况下可作为 MASTER
AC-2：业务节点，正常情况下可作为 STANDBY 或接管 MASTER
AC-3：仲裁/见证节点，参与 Hazelcast 多数派，不响应 DHCP
```

核心规则：

- 用户、AP、业务终端通过 VIP 访问 AC 服务。
- 当前 MASTER 才能响应 DHCP。
- STANDBY 只同步状态，不对外发送 DHCP 响应。
- AC-3 只做仲裁和租约副本，不承载 VIP。
- 无法确认多数派或租约状态时，系统拒绝新 DHCP 分配。

## 2. HTTP/API 访问

HTTP/API 通过 VIP 暴露。OpenResty 部署在 AC-1 和 AC-2 本机，将请求代理到本机 Spring Boot。

日常访问方式：

```text
http://<VIP>:<HTTP_PORT>/
```

切换时的预期行为：

- MASTER 故障后，VIP 漂移到 STANDBY。
- OpenResty 在新 MASTER 本机继续代理 HTTP/API。
- 短时间访问失败可以接受，但应自动恢复。
- 恢复旧主后，VIP 不应频繁来回漂移。

## 3. DHCP 热备行为

DHCP 客户端仍按标准 DHCP 流程请求地址。两台业务 AC 都可能收到广播，但只有 MASTER 会响应。

期望行为：

| 场景 | 结果 |
|---|---|
| 正常运行 | 只有 MASTER 返回 `DHCPOFFER` 和 `DHCPACK`。 |
| STANDBY 收到 DHCP 广播 | 丢弃请求，不响应。 |
| MASTER 故障 | STANDBY 接管 VIP 后成为 MASTER，再响应 DHCP。 |
| Hazelcast 无多数派 | 禁止新分配。 |
| 旧主恢复 | 先同步租约，不直接响应 DHCP。 |

DHCP Option 54 Server Identifier 应为 VIP。抓包时如果看到物理 IP，说明客户端续租可能绕过 VIP，需要修正配置项 `dhcp.server-identifier`。

## 4. 租约同步与恢复

租约状态由 Hazelcast `dhcp:leases` 维护，H2 `dhcp_lease` 保存本地镜像。

租约分配时：

```text
客户端请求
  -> MASTER 检查 HA 条件
  -> 查询或选择租约
  -> 写入 Hazelcast
  -> 写入本地 H2 镜像
  -> 发送 DHCP 响应
```

节点启动时：

```text
读取本地 H2 镜像
加入 Hazelcast 集群
同步或合并租约视图
镜像加载完成后才允许接管
```

运维观察重点：

- STANDBY 是否有完整租约视图。
- 主备切换后是否重复分配旧 IP。
- 两台本地 H2 是否出现长期状态漂移。
- Hazelcast 成员数是否低于多数派阈值。

## 5. 健康检查与状态接口

Keepalived 通过健康检查决定是否继续持有 VIP。健康检查应覆盖应用状态、HA 角色、VIP、Hazelcast 和租约镜像。

建议接口：

```text
POST /internal/ha/role?role=MASTER|BACKUP|FAULT
GET  /actuator/health/ha
GET  /internal/ha/status
```

`notify.sh` 在 Keepalived 状态变化时调用角色接口。`check_app.sh` 调用 HA 健康接口，如果应用不具备服务能力，应让 Keepalived 释放 VIP。

## 6. 广播、多播与定时任务

广播、多播和定时任务容易在主备同时运行时产生重复副作用。所有会影响外部设备的行为都必须受 HA 角色控制。

MASTER 可执行：

- 设备发现广播。
- 主动控制类广播/多播。
- 对 AP/业务终端的指令下发。
- DHCP 租约过期回收。
- 会写入业务状态的定时任务。

STANDBY 可执行：

- 租约同步。
- 本地镜像更新。
- 健康检查。
- 只读监控。

如果抓包看到 AC-1 和 AC-2 同时发送相同广播或多播，说明存在应用层双主风险。

## 7. 常用验证命令

查看 VIP：

```bash
ip addr show
```

查看 Keepalived 状态：

```bash
systemctl status keepalived
journalctl -u keepalived -f
```

验证 HTTP/API：

```bash
ping <VIP>
curl -i http://<VIP>:<HTTP_PORT>/
```

抓 DHCP 包：

```bash
tcpdump -i <网卡> -nn -vvv -s 0 -w dhcp-ha.pcap 'udp port 67 or udp port 68'
```

Wireshark DHCP 过滤：

```text
udp.port == 67 or udp.port == 68
```

抓广播包：

```bash
tcpdump -i <网卡> -nn -vvv -s 0 -w broadcast-ha.pcap 'broadcast'
```

Wireshark 广播过滤：

```text
eth.dst == ff:ff:ff:ff:ff:ff
```

抓多播包：

```bash
tcpdump -i <网卡> -nn -vvv -s 0 -w multicast-ha.pcap 'multicast'
```

查看多播组：

```bash
ip maddr show
netstat -g
```

## 8. 验证观察项

VIP 与 HTTP/API：

- 两台 AC 同时启动后，是否只有一台持有 VIP。
- 停止 MASTER Keepalived 后，VIP 是否漂移到 STANDBY。
- 切换后 HTTP/API 是否自动恢复。
- 恢复旧主后 VIP 是否稳定，不频繁漂移。

DHCP：

- 同一次 DHCPDISCOVER 是否只收到一个 DHCPOFFER。
- 同一次 DHCPREQUEST 是否只收到一个 DHCPACK。
- Option 54 Server Identifier 是否为 VIP。
- 主备切换后新客户端是否拿到旧客户端已占用 IP。
- Hazelcast 断开时是否仍继续分配新 IP。

广播/多播：

- 同一周期是否 AC-1、AC-2 各发一份相同消息。
- 源 IP 是 VIP 还是物理 IP。
- STANDBY 是否处理并写入业务状态。
- 旧主恢复后是否继续发送业务网络包。
