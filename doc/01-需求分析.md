# AC 两节点 AP 优先高可用需求分析

## 1. 背景

当前系统是 AC 产品，负责管理 AP、业务终端和边缘设备。边缘设备主要包括遥测仪器、穿戴设备等，设备用 SN 标识，运行时需要记录 `SN -> IP` 的当前映射。AC 内置 DHCP 服务负责地址分配，并通过 HTTP/API 暴露管理能力。

第一版主方案采用两台业务 AC 部署，目标是减少过度高可用设计：不引入第三节点仲裁，不使用 Keepalived 作为正式 VIP 漂移组件，不通过 Hazelcast 多数派阻断业务。系统默认运行在 Ubuntu。

## 2. 高可用目标

- 主备只由本机是否实际持有 VIP 判断：本机有 VIP 就是主，本机没有 VIP 就是备。
- `application.yml` 增加 `ha.ip`；配置后作为 VIP，未配置时默认取 Ubuntu 第一块物理网卡 IPv4。
- 网络分区时偏向 AP，允许两台 AC 在各自分区内同时持有 VIP 并继续服务。
- 分区恢复后先收敛 VIP，再以当前 VIP 持有节点为主视图合并数据。
- DHCP、边缘设备、AC 配置都以本地 H2 为持久化镜像，以 Hazelcast 为在线同步与恢复输入。
- 配置以 `application.yml` 为初始默认值，可通过 API 修改并持久化到 H2；下次启动时 H2 覆盖默认值。

## 3. 同步数据范围

Hazelcast 第一版同步三类数据：

| Map | Key | Value | H2 镜像 |
|---|---|---|---|
| `dhcp:leases` | IP | DHCP 租约记录 | `dhcp_lease` |
| `edge:devices` | 边缘设备 SN | 边缘设备 IP | `edge_device` |
| `ac:config` | 固定 `effective` | 生效配置 JSON | `ac_config` |

`edge:devices` 中 SN 统一 trim 后作为 key，IP 必须是合法 IPv4。最终视图中 IP 按唯一处理，避免同一个 IP 指向多个 SN。

## 4. 分区恢复要求

- 恢复触发条件为 Hazelcast 成员从单节点可见恢复到双节点可见，或周期任务发现双节点可见。
- VIP 收敛后，实际持有 VIP 的节点为主视图。
- DHCP 租约发生同 IP 或同 MAC 冲突时，保留主侧租约；弃用租约后续请求返回 DHCPNAK 等消极通知，不建立独立审计表。
- 边缘设备同 SN 不同 IP、或同 IP 不同 SN 时，保留主侧记录；非冲突记录补并到主视图。
- AC 配置按完整资源处理，恢复时以主侧 `ac_config.effective` 整体覆盖非主侧，不做字段级合并。

## 5. API 要求

- `GET /internal/ac/config` 查询当前生效 AC 配置。
- `PUT /internal/ac/config` 完整替换 DHCP + HA 相关配置，仅允许 VIP 持有节点写入。
- `GET /internal/edge-devices` 查询边缘设备 SN/IP 列表。
- `GET /internal/edge-devices/{sn}` 查询单个边缘设备。
- `PUT /internal/edge-devices/{sn}` 写入 `{ "ip": "192.168.x.x" }`，仅允许 VIP 持有节点写入。

## 6. 成功标准

- Hazelcast 成员不足时，DHCP 不因多数派不足被阻断。
- 本机持有 VIP 且配置、租约镜像加载完成时可以响应 DHCP。
- 分区期间两侧可独立写入 DHCP 租约、边缘设备和 AC 配置。
- 分区恢复后只保留一个 VIP 持有节点，并按主侧优先策略收敛三类数据。
- API 修改后的 AC 配置持久化成功，重启后按 H2 配置启动。
