# AC 高可用与 DHCP 热备需求分析

## 1. 业务背景

当前系统为 AC 产品。AC 下管理多个 AP，AP 下管理多个 CMS，AP 和 CMS 都需要 IP 地址，由 AC 内置 DHCP 服务完成地址分配。

系统部署在封闭医疗内网中。医院原有设备 IP 基本固定，接入本医疗设备系统后，现场主要配置 AC 的业务 IP、DHCP 地址池与预留 IP。该场景对地址冲突非常敏感：短时间无法获取新 IP 可以接受，但重复分配 IP 可能导致 AP/CMS 连接异常、设备绑定混乱或业务中断。

## 2. 当前系统现状

- Spring Boot 已实现 AC 基本业务功能。
- 项目已初步引入 Hazelcast，按集群模式在 AC 节点之间通信。
- DHCP 服务由 Netty 与 Java Net 实现。
- DHCP 租约当前使用 `ConcurrentHashMap` 本地内存保存。
- 其他缓存模块主要使用 Caffeine。
- 数据库为 H2 文件模式，各 AC 节点使用本地数据库文件，不共享数据库。
- ORM 使用 Hibernate，并引入 Spring Data JPA。
- 数据库初始化已使用 Flyway。
- DHCP 默认使用当前主机第一块物理网卡 IP 作为 Server IP。
- DHCP 服务默认随应用自动启动，除非在管理界面人工停用。

## 3. 高可用目标

第一阶段主方案采用“三节点仲裁”：

```text
AC-1：业务节点，Spring Boot + DHCP + OpenResty + Keepalived + Hazelcast
AC-2：业务节点，Spring Boot + DHCP + OpenResty + Keepalived + Hazelcast
AC-3：仲裁/见证节点，Hazelcast 数据成员，可选运行监控，不承载 VIP 和 DHCP
```

目标如下：

- AC-1 与 AC-2 部署在同一业务二层网段内，形成业务主备。
- AC-3 参与 Hazelcast 集群、租约副本和多数派判断。
- AC 管理端 HTTP/API 通过 VIP 暴露，VIP 默认只在 AC-1 与 AC-2 之间漂移。
- DHCP 服务随主备切换恢复工作，同一时刻只允许当前 MASTER 响应。
- STANDBY 可以接收租约同步，但不发送 `DHCPOFFER`、`DHCPACK` 或其他业务响应。
- 任一节点不能确认自己持有 VIP、不能确认 Hazelcast 多数派、不能确认租约状态可信时，必须拒绝新 DHCP 分配。
- 主备切换期间允许短暂无法获取 IP，但不允许重复分配 IP。
- 尽量保持单体和轻量，优先复用 Spring Boot、Hazelcast、H2、Flyway、OpenResty 与 Keepalived。

## 4. 部署与网络约束

- AC-1、AC-2、AC-3、AP、CMS 位于同一个业务网络或可互通的内网环境。
- AP/CMS 的 DHCP 广播能被 AC-1 与 AC-2 同时收到。
- 没有额外交换机、路由器或独立网关服务器可用于部署统一 Nginx。
- OpenResty 只能部署在业务 AC 本机，用于 HTTP/API 反向代理，不处理 DHCP。
- AC 部署服务器性能一般，方案不能引入过重依赖。
- VIP、AC-1 物理 IP、AC-2 物理 IP、AC-3 物理 IP、抓包 PC IP 必须从 DHCP 地址池中排除。

## 5. 技术与数据约束

- H2 是本地文件模式，不是跨节点共享数据库。
- 当前租约只在本地内存中，不满足 HA 场景的权威状态要求。
- IP 池主要依赖手动配置范围与预留 IP。
- OpenResty 只能解决 HTTP/API 入口问题，不能解决 DHCP 高可用。
- DHCP 是 UDP 67/68 广播协议，高可用核心在于 VIP 主备、DHCP 单主响应、租约一致性和网络副作用控制。
- Hazelcast `dhcp:leases` 应作为运行时权威租约状态，H2 `dhcp_lease` 应作为各业务节点的本地持久化镜像。

## 6. 风险与优先级

双节点主备最大风险是没有第三方仲裁。网络分区时，AC-1 与 AC-2 可能都认为自己应继续服务，最终出现双主响应、重复 DHCP 分配或本地租约状态漂移。

三节点仲裁的核心价值不是增加一个 DHCP 服务，而是引入多数派：

```text
多数派 = 至少 2 个 Hazelcast 成员可见
AC-1 + AC-3：允许 AC-1 所在侧服务
AC-2 + AC-3：允许 AC-2 所在侧服务
AC-1 单独成孤岛：禁止 AC-1 分配新 IP
AC-2 单独成孤岛：禁止 AC-2 分配新 IP
```

风险处理优先级：

1. 优先避免重复分配 IP。
2. 其次保证 HTTP/API 与 DHCP 在主备切换后自动恢复。
3. 再处理切换时间、状态页面、自动化验证和运维体验。

## 7. 成功标准

- 同一时刻只有一个节点持有 VIP。
- 同一时刻只有当前 MASTER 响应 DHCP 请求。
- DHCP Option 54 Server Identifier 使用 VIP，不使用单机物理 IP。
- Hazelcast 少于多数派时，拒绝新 DHCP 分配。
- STANDBY 接管前已具备可信租约视图，不从空内存重新分配地址池。
- 广播、多播、定时任务等对外有副作用的行为只由 MASTER 执行。
- 旧主恢复后先作为 STANDBY 同步租约，不抢占，不继续发送业务网络包。
- 验证材料能通过抓包、日志、接口和数据库记录证明上述行为。
