# AC 两节点 AP 优先高可用详细设计

## 1. 主备判定

运行时不维护独立角色状态，只读本机网卡：

| 判断 | 含义 |
|---|---|
| `hasVip=true` | 本机是主，可在配置和租约镜像就绪后响应 DHCP、处理写入 API。 |
| `hasVip=false` | 本机是备，不响应 DHCP，不执行对外副作用写入。 |

状态接口可以展示由 `hasVip` 推导出的角色标签，但不能把该标签作为单独状态机使用。

## 2. VIP 来源

VIP 选择顺序：

1. `ha.ip` 配置了有效 IPv4 时，使用该 IP。
2. `ha.ip` 为空时，从 Ubuntu 第一块 up 状态、非 loopback、非 virtual、带 IPv4 的物理网卡获取地址。
3. `ha.interface-name` 已配置时，只从该网卡解析默认 VIP 并执行 VIP 检查。
4. 无法解析有效 VIP 时，本节点不响应 DHCP，并在状态接口和健康检查中暴露。

VIP 控制第一版可使用：

```bash
ip address add <VIP>/<PREFIX> dev <IFACE>
ip address del <VIP>/<PREFIX> dev <IFACE>
arping -U -c 3 -I <IFACE> <VIP>
```

协调规则：

- Hazelcast 只看到自己时，本节点确保 VIP 绑定。
- Hazelcast 看到两个 AC 时，按 `cluster.members` 第一项保留 VIP；未配置固定成员时按可见成员地址排序选择第一项。
- 非保留节点释放 VIP。
- `ha.vip-management-enabled=false` 只用于本地测试或临时验证，正式 Ubuntu 部署默认开启。

## 3. DHCP 服务闸门

`canServeDhcp()` 判定条件：

```text
dhcp.enabled == true
configLoaded == true
leaseMirrorLoaded == true
hasVip == true
```

Hazelcast lifecycle 和成员数只作为状态、告警与恢复合并输入，不因成员不足拒绝 DHCP。

## 4. Hazelcast Map

| Map | Java 语义 | 说明 |
|---|---|---|
| `dhcp:leases` | `IP -> DhcpLeaseRecord` | 在线租约视图。 |
| `edge:devices` | `SN -> IP` | 边缘设备当前 IP，SN trim 后作为 key。 |
| `ac:config` | `effective -> JSON` | 当前生效 DHCP + HA 配置。 |

这些 Map 不配置 split-brain protection 作为服务闸门。网络分区期间允许各分区独立写入，恢复后由应用层按主侧优先收敛。

## 5. H2 镜像

| 表 | 用途 |
|---|---|
| `dhcp_lease` | 本地 DHCP 租约镜像。 |
| `edge_device` | 本地边缘设备 SN/IP 镜像。 |
| `ac_config` | 本地 AC 配置快照，固定 key 为 `effective`。 |

启动顺序为：

```text
application.yml 默认值
读取 ac_config.effective 并覆盖默认值
发布 ac:config
加载 dhcp_lease 到 dhcp:leases
加载 edge_device 到 edge:devices
标记配置与租约镜像就绪
```

## 6. AC 配置 API

接口：

```text
GET /internal/ac/config
PUT /internal/ac/config
```

`PUT` 使用完整资源替换，范围限定在 DHCP + HA 相关配置。服务端校验：

- IPv4 字段格式。
- DHCP 地址池起止范围。
- 排除地址和保留地址格式。
- `ha.ip` 不得落入 DHCP 动态地址池。
- `ha.prefixLength` 范围为 1 到 32。
- `ha.commandTimeoutSeconds` 至少为 1 秒。
- 租期下限为 60 秒。
- 端口范围为 1 到 65535。

写入顺序为校验、写 H2、应用到运行时配置、发布 `ac:config`。非 VIP 持有节点拒绝写入。

## 7. 边缘设备 API

边缘设备主要是遥测仪器、穿戴设备等，第一版只维护 SN 与 IP：

```text
GET /internal/edge-devices
GET /internal/edge-devices/{sn}
PUT /internal/edge-devices/{sn}
```

`PUT /internal/edge-devices/{sn}` 请求体：

```json
{ "ip": "192.168.56.210" }
```

规则：

- SN trim 后不能为空，最大 128 字符。
- IP 必须是合法 IPv4。
- 最终视图中 IP 唯一。
- 非 VIP 持有节点拒绝写入。
- 写入成功后同步保存到 `edge_device` 和 `edge:devices`。

## 8. 分区恢复

恢复触发：Hazelcast 成员数从 1 恢复到 2，或周期同步发现双节点可见。

通用流程：

```text
先收敛 VIP
读取本机 hasVip
hasVip=true 的节点生成最终视图
hasVip=false 的节点从 Hazelcast 最终视图覆盖本地 H2
```

DHCP 租约：

- 同 IP 不同 MAC，保留主侧。
- 同 MAC 不同 IP，保留主侧。
- 非冲突租约补并。
- 弃用租约不建审计表；后续请求返回 DHCPNAK，引导客户端重新申请。

边缘设备：

- 同 SN 不同 IP，保留主侧 IP。
- 同 IP 不同 SN，保留主侧 SN/IP。
- 非主侧独有且 IP 不冲突的 SN 补并。
- 非 VIP 节点恢复后以 `edge:devices` 覆盖本地 `edge_device`。

AC 配置：

- 以 VIP 持有侧 `ac_config.effective` 为完整权威配置。
- 非 VIP 侧分区期间修改的配置在恢复后被覆盖。
- 不做字段级合并，避免混合配置不可预测。

## 9. 第一版边界

第一版包含 Ubuntu、应用内 VIP 控制、三类 Hazelcast Map、三类 H2 镜像、AC 配置 API、边缘设备 API、DHCP 按 VIP 响应和 AP 优先恢复合并。

第一版不包含第三节点仲裁、Hazelcast CP Subsystem、Keepalived 正式部署、独立冲突审计表、全部 AC 业务配置运行时持久化、多网段 DHCP。
