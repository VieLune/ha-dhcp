# AC 高可用与数据同步项目计划

## 1. 目标

把现有方案收敛为“两节点 AP 优先 + 应用自控 VIP + Hazelcast 三类数据同步 + H2 本地镜像”。正式链路不使用 Keepalived、第三节点仲裁或多数派拒绝写入。

## 2. 阶段计划

### 阶段 1：HA 基线清理

- 移除 DHCP 服务链路中的手动运行时角色和多数派依赖。
- `canServeDhcp()` 改为只看 DHCP 启用、配置加载、租约镜像加载、本机实际持有 VIP。
- `ha.ip` 为空时解析 Ubuntu 第一块物理网卡 IPv4。
- 增加应用内 VIP 协调循环，调用 `ip address add/del` 和 `arping` 绑定、释放、宣告 VIP。

验收：Hazelcast 只看到自己时，只要本机持有 VIP 且本地镜像就绪，就可以响应 DHCP。

### 阶段 2：AC 配置持久化

- 新增 `ac_config` 表。
- 新增 `ac:config` Hazelcast Map。
- 实现 `GET /internal/ac/config` 和 `PUT /internal/ac/config`。
- 启动时按 `application.yml -> H2 -> runtime -> Hazelcast` 顺序加载。

验收：API 修改后持久化成功，重启后 H2 覆盖默认配置。

### 阶段 3：边缘设备同步

- 新增 `edge_device` 表。
- 新增 `edge:devices` Hazelcast Map。
- 实现边缘设备 SN/IP 查询与写入 API。
- 写入只允许 VIP 持有节点执行，IP 在最终视图中唯一。

验收：`PUT /internal/edge-devices/{sn}` 可写入 H2 和 Hazelcast；非 VIP 节点写入被拒绝。

### 阶段 4：分区恢复合并

- DHCP 租约按主侧优先、非冲突补并、冲突弃用处理。
- 边缘设备同 SN 或同 IP 冲突时主侧优先，非冲突 SN 补并。
- AC 配置以主侧完整资源覆盖非主侧。
- 非 VIP 节点恢复后用 Hazelcast 最终视图覆盖本地 H2。

验收：网络恢复后只保留一个 VIP 持有者，三类数据按策略收敛。

## 3. 测试计划

- 启动配置：验证 `application.yml` 默认值、H2 覆盖、`ha.ip` 为空时的默认 VIP 解析。
- DHCP 回归：验证 Hazelcast 成员不足不阻断 DHCP，非 VIP 节点不响应 DHCP。
- AC 配置：验证完整替换、非法 IP/端口/地址池拒绝、重启后配置仍生效。
- 边缘设备：验证 SN trim、IPv4 校验、IP 唯一、非 VIP 写入拒绝。
- 分区恢复：验证 DHCP 租约冲突、边缘设备同 SN/同 IP 冲突、AC 配置冲突的主侧优先策略。

## 4. 风险

| 风险 | 影响 | 处理 |
|---|---|---|
| VIP 控制权限不足 | 无法自动接管 | Ubuntu 运行账号授予 root 或 `CAP_NET_ADMIN`。 |
| 默认网卡识别错误 | VIP 选错地址 | 生产环境显式配置 `ha.ip` 和 `ha.interface-name`。 |
| 分区期间双主写入冲突 | 数据恢复后部分记录被弃用 | 主侧优先，非冲突补并，冲突按协议消极通知。 |
| Hazelcast 合并后丢失非主侧部分临时视图 | 非冲突补并不完整 | 依赖本地 H2 镜像和周期同步逐步收敛；关键场景通过演练验证。 |
| Keepalived 文件误用 | 部署认知混乱 | deploy 文档标明仅为历史验证材料。 |

## 5. 完成标准

- 文档、代码和配置统一为两节点 AP 优先方案。
- Hazelcast 含 `dhcp:leases`、`edge:devices`、`ac:config`。
- H2 含 `dhcp_lease`、`edge_device`、`ac_config`。
- AC 配置与边缘设备 API 可用。
- 测试覆盖 HA 判定、配置校验、边缘设备写入和 DHCP 回归。
